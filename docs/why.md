# Why Contextual

> **Contextual — The Developer's New Home**

## The Problem

The IDE was the developer's home for decades. One place, one focus, everything you needed.

That era is over.

In the age of AI, the modern developer's workflow is a mess of fragmented tools:

- **Terminal** — running Claude Code, watching logs, managing processes
- **Linear** — reading tickets, updating statuses, chasing context
- **Figma** — cross-referencing designs mid-implementation
- **Slack** — digging through threads for decisions that were never documented
- **IDE** — actually writing code (less and less)

The IDE hasn't evolved. It still thinks the repo is the unit of work. But features don't live in a single repo anymore — they span frontend, backend, infrastructure, and design simultaneously.

## The Deeper Issue

Two things broke the old model:

### 1. The repo is the wrong unit of work
A feature touches multiple repos. Working on FE alone or BE alone is artificial — the feature doesn't care about your repo boundaries. Your tools should reflect that.

### 2. AI unlocks parallelism that has no home
If AI can handle most of the implementation, a developer can run 3, 5, 10 features in parallel. But there's no tool designed for that reality. You end up with 15 terminal tabs, no structure, and no visibility.

## The Result

Developers spend more time managing context than building. The cognitive overhead of context-switching between tools is the new bottleneck — not typing speed, not syntax knowledge.

**The developer's home needs to be rebuilt from scratch.**
